## Context

The agent handles sensitive expert topics (medicine, psychology) and already has:
- **Emotional Analyzer** with Circumplex model (valence/arousal), quadrant classification (`distress`, `melancholy`, etc.), and trend tracking via a sliding window.
- **Prompt system** with a `crisis_prompt` layer that provides soft LLM guidance for crisis situations.
- **Data channel** infrastructure for publishing typed events to the client.
- **`crisis_contacts` table** referenced in the original schema design but not yet created.

The current crisis handling is purely advisory — the LLM may ignore or dilute the crisis prompt when other context (RAG results, conversation history) competes for attention. S19 introduces a deterministic override that guarantees a safe response.

## Goals / Non-Goals

**Goals:**
- Detect distress signals in user utterances with high recall (minimize false negatives)
- Override the normal LLM+RAG pipeline with a fixed empathetic response protocol
- Provide locale-aware emergency contacts from the database
- Log crisis events for audit and monitoring
- Notify the client via data channel so the UI can render emergency contacts

**Non-Goals:**
- Clinical diagnosis or risk assessment scoring (out of scope, liability concern)
- Persistent crisis state across sessions (each session is independent)
- Client-side crisis detection (all detection happens server-side in the agent)
- Automated external notifications (e.g., calling emergency services)
- Content moderation of non-crisis inappropriate content

## Decisions

### 1. Two-tier detection: keyword fast-path + LLM classification

**Decision:** Use a two-tier detection approach. Tier 1 is a fast keyword/pattern matcher that runs synchronously on every utterance. Tier 2 is an LLM-based classifier invoked only when Tier 1 produces a match or when the emotional analyzer signals sustained distress.

**Rationale:** Pure keyword matching has high recall but low precision (e.g., "I'm dying of laughter"). Pure LLM classification is accurate but adds latency. The two-tier approach uses keywords as a fast filter and LLM as a confirmation step, keeping latency low for non-crisis utterances (>99% of traffic) while maintaining accuracy for actual crises.

**Alternatives considered:**
- Keyword-only: Too many false positives, would trigger on benign expressions.
- LLM-only on every utterance: Adds ~500ms latency to every turn. Wasteful.
- Fine-tuned classifier model: Higher accuracy but requires training data, maintenance burden, and an additional model dependency.

### 2. Pipeline override via before_llm_cb hook

**Decision:** Insert the crisis detector as a `before_llm_cb` callback in the AgentSession. When a crisis is confirmed, the callback replaces the LLM context with a fixed crisis response template populated with locale-specific contacts, and sets a flag that bypasses RAG injection and dual-layer parsing.

**Rationale:** The LiveKit AgentSession supports `before_llm_cb` which fires before each LLM call. This is the cleanest integration point — no need to fork the pipeline or add conditional logic throughout `agent.py`. The crisis response bypasses RAG to avoid polluting the safety message with potentially harmful knowledge base content.

**Alternatives considered:**
- Post-LLM filtering: Too late — the LLM may already generate harmful content, and we waste compute.
- Separate crisis agent: Over-engineered for a single response pattern. Adds room management complexity.
- Middleware in the pipeline: Would require refactoring the existing pipeline chain.

### 3. Crisis response generated by LLM with constrained prompt, not hardcoded

**Decision:** The crisis response is generated by the LLM but with a tightly constrained system prompt that enforces the protocol (empathy → non-judgment → professional help recommendation → contacts). The crisis prompt includes the actual contact details fetched from the database. Conversation history is excluded from context during crisis override.

**Rationale:** A hardcoded response feels robotic and cannot adapt to the user's language, emotional tone, or specific situation. Using the LLM with a constrained prompt produces natural, empathetic responses while the rigid prompt structure ensures all required elements are present. Excluding conversation history prevents the LLM from being influenced by prior context that might dilute the crisis response.

**Alternatives considered:**
- Fully hardcoded template: Feels cold, cannot adapt to language/tone, needs separate templates per language.
- Normal LLM with boosted crisis prompt: The LLM can still be influenced by other context. Not deterministic enough.

### 4. Crisis contacts stored in database, cached per session

**Decision:** Emergency contacts are stored in the `crisis_contacts` table with language/locale and contact type. On session start, contacts for the session's locale are fetched and cached. During a crisis event, cached contacts are used (no DB query in the hot path).

**Rationale:** DB storage allows updating contacts without code deployment. Session-level caching avoids latency during the critical crisis response moment. The contact set is small (typically 3-5 per locale) so memory cost is negligible.

### 5. Crisis event logging via messages table with is_crisis flag

**Decision:** Add an `is_crisis` boolean column to the `messages` table. When a crisis is detected, both the user's triggering message and the agent's crisis response are flagged. Additionally, a `crisis_alert` event is published via data channel.

**Rationale:** Using the existing `messages` table keeps the audit trail in the same place as all other conversation data. The `is_crisis` flag enables simple queries for monitoring dashboards. A separate `crisis_events` table was considered but adds schema complexity for minimal benefit at MVP scale.

### 6. Emotional analyzer integration: distress signal as crisis input

**Decision:** When the emotional analyzer detects sustained `distress` quadrant (3+ consecutive snapshots with falling valence trend), it sets a `high_distress` flag that lowers the crisis detector's keyword matching threshold (more sensitive) and can independently trigger Tier 2 LLM classification even without keyword matches.

**Rationale:** Some crisis situations escalate gradually without explicit keywords. A user saying "I just can't anymore... nothing works... I'm so tired of everything" may not trigger keyword detection but shows a clear distress pattern. The emotional trend provides a complementary signal.

## Risks / Trade-offs

- **False positives** → The two-tier system mitigates this. Tier 1 keywords are intentionally broad, but Tier 2 LLM classification filters out benign usage (e.g., "this exam is killing me"). If the LLM classifier is uncertain, it defaults to triggering (false positive preferred over false negative for safety).

- **False negatives with novel language** → Keyword lists cannot cover all expressions of distress across languages and cultures. Mitigation: the emotional analyzer's distress signal provides a keyword-independent detection path. The keyword list is stored in the database (via agent_configs) for easy updates.

- **LLM classifier latency during crisis** → Tier 2 adds ~300-500ms. Mitigation: acceptable for safety-critical path. The crisis response generation itself is fast because context is minimal (no RAG, no history).

- **Liability concerns** → The system explicitly does NOT diagnose, assess risk levels, or promise professional help. It recommends contacting professionals and provides contact information. This is documented in the crisis prompt.

- **Crisis prompt override disrupts conversation flow** → After a crisis response, the agent returns to normal mode. If the user continues expressing distress, the detector triggers again. There is no "crisis session lock" — this is intentional to avoid trapping users in a loop if a false positive occurs.

- **Multi-language keyword coverage** → Initial seed covers Russian and English. Mitigation: keyword lists are config-driven (stored in agent_configs), easily extensible without code changes.
