## Context

Аутентификация (S03) реализована: JWT access/refresh токены, middleware `get_current_user`. Модели `Session` и `Message` существуют в БД (S02). LiveKit Server запущен в Docker (S01), переменные `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET`, `LIVEKIT_URL` уже определены в `.env.example`.

Клиенту нужен LiveKit access token для подключения к комнате. Токен генерируется на стороне API и содержит права участника. Без этого эндпоинта голосовая сессия невозможна.

## Goals / Non-Goals

**Goals:**
- Эндпоинт создания сессии с генерацией LiveKit токена
- Эндпоинты просмотра истории сессий и сообщений
- Авторизация через существующий JWT middleware
- Тесты для всех эндпоинтов

**Non-Goals:**
- Завершение сессии (будет управляться агентом в S05+)
- Снимок конфигурации агента (`agent_config_snapshot`) — заполняется в S10
- WebSocket/realtime уведомления о статусе сессии
- Пагинация на основе курсора (достаточно offset/limit для MVP)

## Decisions

### 1. Генерация room_name

**Решение:** Формат `session-{session_uuid}` — детерминированный, уникальный, легко сопоставить с записью в БД.

**Альтернатива:** Случайные слова (`adjective-noun-1234`) — красивее, но сложнее маппить обратно к сессии.

### 2. LiveKit SDK для генерации токенов

**Решение:** Использовать `livekit-api` (официальный Python SDK). Класс `AccessToken` + `VideoGrants` с правами `room_join`, `room`, `can_publish`, `can_subscribe`, `can_publish_data`.

**Альтернатива:** Ручная генерация JWT — ненадёжно, нужно отслеживать формат токена LiveKit вручную.

### 3. Структура модуля

**Решение:** Роутер `src/sessions/router.py`, сервис `src/sessions/service.py`, схемы `src/schemas/sessions.py`. Сервис LiveKit-токенов в `src/sessions/livekit.py` — изолирован для переиспользования агентом в будущем.

**Альтернатива:** Всё в одном файле роутера — быстрее, но хуже разделение ответственности при росте.

### 4. TTL токена LiveKit

**Решение:** TTL = 6 часов. Сессия может длиться долго (терапевтическая беседа), при этом токен не должен жить бесконечно. LiveKit сам переподключит клиента при истечении.

### 5. Права доступа к сессиям

**Решение:** Пользователь видит только свои сессии. Фильтрация `WHERE user_id = current_user.id` на уровне запроса. Для `GET /sessions/{id}/messages` — проверка принадлежности сессии пользователю, 404 если чужая.

### 6. Конфигурация LiveKit

**Решение:** Переменные окружения `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET` читаются через Pydantic Settings. `LIVEKIT_URL` не нужен для генерации токена (токен подписывается ключом, URL знает клиент).

## Risks / Trade-offs

- **[LiveKit SDK совместимость]** → Зафиксировать минимальную версию `livekit-api>=1.0.0` в зависимостях. SDK стабилен.
- **[Длинные сессии и TTL токена]** → 6 часов покрывает MVP. В будущем можно добавить эндпоинт обновления токена.
- **[Нет rate limiting на создание сессий]** → Пользователь может создать много сессий. Для MVP допустимо — rate limiting добавится позже на уровне Caddy или middleware.
