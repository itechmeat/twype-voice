## ADDED Requirements

### Requirement: Service worker registration
The application SHALL register a service worker on startup using `vite-plugin-pwa` with `registerType: "prompt"`. The workbox configuration SHALL NOT include `skipWaiting: true` or `clientsClaim: true`. The Service Worker SHALL wait for user confirmation before activating a new version.

#### Scenario: Service worker registers on first visit
- **WHEN** a user visits the app for the first time
- **THEN** the service worker is registered and begins precaching the app shell

#### Scenario: New deployment does not force-reload
- **WHEN** a new Service Worker version is deployed
- **AND** a user has an active LiveKit session
- **THEN** the current Service Worker continues serving the active session
- **AND** the new version waits in the `waiting` state until the user confirms the update

#### Scenario: Update activates on user confirmation
- **WHEN** the user confirms the update via the UpdatePrompt component
- **THEN** the waiting Service Worker activates and the page reloads

### Requirement: App shell precaching
The service worker SHALL precache the application shell — the HTML entry point, CSS bundles, JavaScript bundles, and font files generated by Vite build. The precache manifest SHALL be automatically generated by `vite-plugin-pwa` using content hashes for cache busting.

#### Scenario: App loads offline after initial visit
- **WHEN** a user has previously visited the app and then loses network connectivity
- **THEN** the app shell loads from the service worker cache and displays a functional UI with an offline state indicator

#### Scenario: Precache updates on new deploy
- **WHEN** a new build is deployed with changed JS/CSS bundles
- **THEN** the service worker downloads the updated assets and the next page load uses the new version

### Requirement: Runtime caching for API calls
The service worker SHALL use a **NetworkFirst** strategy for requests to `/api/*` with a network timeout of 3 seconds and a cache fallback. This ensures fresh data when online and cached responses when briefly offline.

#### Scenario: API call succeeds online
- **WHEN** the app makes an API call and the network is available
- **THEN** the response comes from the network and is stored in the runtime cache

#### Scenario: API call falls back when offline
- **WHEN** the app makes an API call that was previously cached and the network is unavailable
- **THEN** the cached response is returned

### Requirement: Static asset caching
The service worker SHALL use a **CacheFirst** strategy for static assets in `/assets/*` (Vite-hashed output). Since these files are content-addressed, cached versions are always valid.

#### Scenario: Hashed assets served from cache
- **WHEN** the browser requests a previously loaded hashed asset (e.g., `/assets/index-abc123.js`)
- **THEN** the asset is served from the cache without a network request
